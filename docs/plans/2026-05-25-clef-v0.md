**Status:** Implemented ✅ · **Planned:** 2026-05-25 · **Completed:** 2026-06-07
**Delivers:** `clef v0.0.1` — first release (`--version` only), shipped via a private Homebrew tap with tag-triggered CD.
**Outcome:** All five acceptance criteria met; the full release loop (tag → CI → tap formula bump → `brew install`/`upgrade`) was verified end-to-end. Per-step checkboxes are left unchecked to avoid churn — the authoritative record is the **Acceptance criteria** (all five checked) plus **Execution notes — deviations & decisions**, both at the end of this file.

> **For agentic workers:** Implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `clef v0.0.1` — a bb-runnable script that prints `--version`, distributed via a private Homebrew tap (with `babashka` declared as a brew dep), end-to-end CD running on every tag push.

**Architecture:** A single self-contained Clojure script is generated by `bb gen-script` from sources under `src/`, committed to repo root as `./clef`. Pushing a `v*` tag triggers one GitHub Actions job that updates the formula in private `kirahowe/homebrew-clef` with the tag's GitHub archive zip URL + SHA256. No matrix build, no release artifacts, no GraalVM. Directly modeled on [`neil`](https://github.com/babashka/neil).

**Tech Stack:** Clojure, babashka, GitHub Actions, Homebrew, MIT license.

**Scope:** This plan implements *only* v0.0.1. Do not drift into `clef new`, `clef deploy`, or any v0.0.2+ work. Full design rationale, layout seams, roadmap, and deferred items live in the **Design context** section below (consolidated from the former standalone design spec).

## Design context

### Vision (north star, not v0)

`clef` is an opinionated, Rails-inspired full-stack web framework for Clojure. Long-term: `clef new myapp && clef deploy` produces a deployed Clojure web app in under a minute, with no babashka or JVM-tooling literacy required from the developer. The full design vision (plug-in manifest, slot system, multi-template support, per-library composability, Datalevin/Postgres dual paradigms, full-stack Hiccup template) is the north star — **this document is not that.** This is the v0 cut: the smallest thing that proves the CD pipeline works end-to-end. Everything else accretes from here.

The user is the sole consumer of v0. The binary won't be useful to anyone else for a while; the goal is to establish the release engine and the working habits, then add capability behind a working pipeline one command at a time.

### v0 scope

**One command:** `clef --version` prints a version string. That is the entire user-facing surface of the first release.

**Script & build**
- Distribution artifact is a single **bb (babashka) script**, not a GraalVM-compiled native binary — the path well-trodden bb tools like [`neil`](https://github.com/babashka/neil) actually take. `bb gen-script` assembles the namespaces under `src/clef/` into one self-contained file committed to repo root as `./clef`.
- `babashka` is declared as a Homebrew dependency, so `brew install clef` transparently installs bb first if absent. The "no babashka prereq" goal is met via brew's dep resolution rather than custom-build.
- No GraalVM, no per-platform build matrix. One script, runs everywhere `bb` runs. The native-binary-via-babashka-custom-build path is **deferred indefinitely** (real but uncommon, weeks of work, no user-visible benefit while brew handles transitive deps).

**Distribution**
- Main `clef` repo: **public** from day 1 — CD runs in the open.
- Tap repo `kirahowe/homebrew-clef`: **private** for the whole v0 phase. The owner can `brew tap kirahowe/clef && brew install clef`; strangers cannot. Flips public at v1.
- No `install.sh`, `scoop`, `winget`, or `bbin` channels in v0. The script is `bbin`-installable for free (bb.edn carries `:bbin/bin` metadata), but advertising that waits until there's a reason.

**Release pipeline**
- Trigger: pushing a `v*` git tag. The tag is the intent-to-release signal.
- On tag push, one GitHub Actions job: (1) validates the tag matches `version.clj` and that `./clef` is up to date; (2) computes the SHA256 of GitHub's auto-generated archive zip for the tag; (3) pushes a formula update (new `version` / `url` / `sha256`) to the private tap. No release artifacts uploaded — the formula points at GitHub's archive zip, which contains the committed `./clef`.

**Success criteria (the whole loop, no manual steps):** push a commit bumping the version, watch Actions go green, then on the mac run `brew upgrade clef && clef --version` and see the new version.

**Why this size:** it's the irreducible release engine. Once it exists, every future capability is "add one function + a CLI route." No future feature requires changing *how* releases happen.

### Seams that matter (chosen so v0.0.2+ is cheap)

1. **Command dispatch in `main.clj`** is a dispatch table keyed by command name, even with one command. Adding `new` is "add an entry + a namespace." Stays hand-rolled — no external CLI-parser library. The dispatch surface (`--version` + usage) is ~20 lines; a parser dep would be larger than what it replaces, and any external dep punctures the self-contained single-file `./clef` (the script depends on nothing but `bb` itself; see Tasks 5 & 7). If the command tree grows enough to want a real parser, reach for **`babashka.cli`** first — it ships *inside* babashka, so it keeps the script self-contained. Only consider an external lib (e.g. `lambdaisland/cli`, a dep-free, bb-compatible subcommand parser with nicer auto-help) if its ergonomics clearly justify both the external dep and vendoring its source into `gen-script`. Evaluated and rejected for now (2026-06-04).
2. **`version.clj` as single source of truth.** Read by the script (for `--version`) and by `release.yml` (for tag validation). Avoids the "two places to bump" trap.
3. **`resources/clj/new/clef/`** sits empty in v0 (a `.gitkeep`) but is where mustache scaffold templates live when `clef new` arrives. The path follows clj-new's fixed convention so the same files serve both consumers — see **clj-new compatibility** below.
4. **`dev/clef/gen_script.clj`** keeps script-assembly logic out of `src/`. The generated `./clef` is committed so the formula install (`bin.install "clef"`) just works.
5. **No fast-path / JVM-required split yet.** Everything is bb-side (fast-path) at v0. The seam (a "command category" field in the dispatch table) gets added with the first command that needs to shell out to `clj`.

### Explicitly NOT in v0

Mustache templating engine in deps; scaffolded project templates; plug-in manifest reading; bb tasks for *generated* projects; `Dockerfile` / `fly.toml` / any deploy code; GraalVM-native binary via babashka custom-build (deferred indefinitely).

### Roadmap (loose sketch, not a commitment)

Included only to confirm v0's seams support what's likely next.

- **v0.0.1** — `clef --version`. The release engine. *(this plan)*
- **v0.0.2** — `clef new myapp`. Scaffolds a single-file Ring app (`deps.edn`, one handler, README). No db/auth/Dockerfile. Exercises the `resources/clj/new/clef/` slot and the mustache rendering path.
- **v0.0.3** — `clef new` adds `Dockerfile` + `fly.toml`; `clef deploy` wraps `flyctl deploy`. End-to-end demo loop works.
- **v0.0.4+** — `clef dev` (local server via `clj`), `clef doctor` (env check), and the first JVM-requiring command — when the fast-path / JVM-required dispatch split becomes real.

### Explicitly deferred from the full design vision

Not rejected — long-term vision, just not v0 or near-term:

- Plug-in manifest schema, `:plugin/*` keys, `META-INF/clef/plugin.edn` resource reading
- Slot system (template-bound and universal); template definitions and the template/plug-in distinction; cross-template adapters
- Library split into separate repos (stays monorepo until a real second consumer exists); per-library independent versioning
- The "no breaking changes ever" commitment (applies from 1.0; 0.x reserves the right to break)
- Datalevin / Datomic / Postgres support; auth, secrets, observability, docs concerns
- Daemon / persistent JVM REPL for in-project commands; bb tasks as a public extension surface inside generated projects
- `bbin`, `scoop`, `winget`, `install.sh` distribution channels
- **CLI-configurable scaffolding** (`clef new` prompting for db / auth / etc.) and the **multiple clj-new template variants** that implies (e.g. `resources/clj/new/clef-postgres-hanko/` for a fixed combo, since clj-new can't prompt interactively). Variant naming and whether each combo is a distinct clj-new template are deferred — decide when `clef new` gains real options, not before. v0 reserves only the single `resources/clj/new/clef/` placeholder.
- **Interactive prompts for `clef new`.** Confirmed direction (2026-06-04): `clef new` *will* prompt the user during setup for project options (db, auth, etc.) — simple choices, not free-form. This is a `clef new` (v0.0.2+) concern, not v0. Open decision for when it lands: plain stdin prompts (`read-line` over a printed option list — zero deps, bb-native, fine for "type 1/2/3") vs. a TUI selection component for polished arrow-key menus. The TUI route is the only natural use case for a library like `charm.clj` (Elm-architecture TUI framework with a `list`/selection component, bb-compatible) — evaluate it *then*, weighing whether arrow-key menus justify a full TUI framework over plain prompts. Note: a TUI lib belongs only in the interactive-prompt layer, never in CLI-argument dispatch (that's parsing, not a terminal UI). Also note this diverges the two `clef new` consumers — interactive `clef new` vs. non-interactive `clojure -Tclj-new` (which gets fixed template variants per the **clj-new compatibility** section).

## clj-new compatibility (forward-looking — shapes one layout choice in v0)

The project template that `clef new` will scaffold (v0.0.2 work, **not** built in v0) must also be a valid [clj-new](https://github.com/seancorfield/clj-new) template, so it is publishable to the Clojure ecosystem and usable via:

```
clojure -Tclj-new create :template clef :name yourname/myapp
```

clj-new resolves templates by **fixed paths**, so v0 reserves the right directory now to avoid a later move. The only v0 change this drives is parking the placeholder at `resources/clj/new/clef/.gitkeep` instead of `resources/clef/`. The rest is recorded here as constraints for the v0.0.2 implementer:

- **Template resource files** must live at `resources/clj/new/clef/` — clj-new looks up template resources by the classpath path `clj/new/<template>/`. v0 only creates this dir (with `.gitkeep`); v0.0.2 fills it with the mustache-templated project files.
- **Template definition namespace** will be `src/clj/new/clef.clj` (ns `clj.new.clef`, a `clef` fn taking `[name & args]`, using `clj.new.templates`'s `renderer` / `->files`). Added in v0.0.2, not v0. It is separate from the bb script — `gen-script` only concatenates `src/clef/*`, so this namespace never enters `./clef`.
- **Templating syntax** is mustache/stencil (`{{...}}`), which is both clj-new's engine and clef's stated choice. Keeping the resource files in mustache means they render identically whether driven by clj-new's `renderer` or by `clef new`'s own rendering — `clef new` must pass the same variable names clj-new conventionally provides (e.g. `name`, `namespace`, `sanitized`).
- **Publishing coordinate** will be `net.clojars.kirahowe/clj-template.clef` (clj-new searches Clojars for a `clef/clj-template` artifact). Whether that artifact is published from this monorepo or split into a dedicated template repo is a v0.0.2 decision; either way the resource files stay at `resources/clj/new/clef/`.

`deps.edn`'s `:paths ["src" "resources"]` (Task 2) already puts both `src/clj/new/` and `resources/clj/new/` on the classpath, so no deps change is needed when the template arrives.

- [ ] **Step 3: Implement `src/clef/main.clj`**

```clojure
(ns clef.main
  (:require [clef.version :as version]))

(defn- print-version [_args]
  (println version/version)
  0)

(defn- print-usage [_args]
  (println "Usage: clef <command>")
  (println)
  (println "Commands:")
  (println "  --version, -v   Print version")
  1)

(def commands
  {"--version" print-version
   "-v"        print-version})

(defn dispatch [argv]
  (let [[cmd & rest] argv
        handler (get commands cmd print-usage)]
    (handler rest)))

(defn -main [& argv]
  (System/exit (dispatch (vec argv))))
```

**Notes:**
- `commands` is a dispatch table — the spec's required seam for incremental command additions.
- `dispatch` is the testable boundary (returns exit code); `-main` is the thin wrapper that calls `System/exit`.
- Tests target `dispatch`, not `-main`, so they don't kill the test runner JVM.

**Goal:** Implement `bb gen-script` so it concatenates `src/clef/version.clj` and `src/clef/main.clj` with a shebang prelude and a `-main` invocation, producing an executable `./clef` script.

**Files:**

**Reference:** neil's `dev/babashka/neil/gen_script.clj` (pasted in the planning notes for this session). The pattern is: slurp the source files in dependency order, prepend a prelude, append an invocation line, spit to the script name at repo root.

- [ ] **Step 1: Create `prelude` (at repo root)**


(Single line. No trailing blank lines.)

- [ ] **Step 2: Create `dev/clef/gen_script.clj`**

```clojure
(ns clef.gen-script
  (:require [clojure.string :as str]))

(defn gen-script
  "Concatenate clef's source namespaces into a single bb-runnable script at ./clef.
   Order matters: version.clj must precede main.clj because main requires version."
  []
  (let [prelude     (slurp "prelude")
        version-src (slurp "src/clef/version.clj")
        main-src    (slurp "src/clef/main.clj")
        invocation  "(apply clef.main/-main *command-line-args*)"
        parts       [prelude version-src main-src invocation]
        script      (str/join "\n\n" parts)]
    (spit "clef" script)
    (.setExecutable (java.io.File. "clef") true)
    (println "Wrote ./clef")))
```

## Task 6: CI workflow

**Goal:** On every PR and push to `main`, run tests + lint + verify the committed `./clef` is up to date with `src/`.

**Files:**
- Create: `.github/workflows/ci.yml`

- [ ] **Step 1: Create `.github/workflows/ci.yml`**

```yaml
name: CI

on:
  pull_request:
  push:
    branches: [main]

jobs:
  test-lint-and-script-fresh:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Java
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: '21'

      - name: Install Clojure CLI + bb + clj-kondo
        uses: DeLaGuardo/setup-clojure@13.0
        with:
          cli: latest
          bb: latest
          clj-kondo: latest

      - name: Cache Clojure deps
        uses: actions/cache@v4
        with:
          path: |
            ~/.m2
            ~/.gitlibs
            ~/.deps.clj
          key: ${{ runner.os }}-deps-${{ hashFiles('deps.edn', 'bb.edn') }}

      - name: Run tests
        run: bb test

      - name: Lint
        run: bb lint

      - name: Regenerate script and verify no diff
        run: |
          bb gen-script
          git diff --exit-code -- clef

      - name: Smoke-test generated script
        run: ./clef --version
```

The "no diff" step is the discipline: if a contributor edits `src/` but forgets to regenerate `./clef` and commit it, CI fails.

- [ ] **Step 2: Push a feature branch and open a draft PR**

## Task 8: Release workflow — tag-triggered formula bump

**Goal:** On `v*` tag push, validate the tag, compute the GitHub archive zip's SHA256, and push a formula update to the private tap.

**Files:**
- Create: `.github/workflows/release.yml`

- [ ] **Step 1: Create `.github/workflows/release.yml`**

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Java + Clojure + bb
        uses: DeLaGuardo/setup-clojure@13.0
        with:
          cli: latest
          bb: latest

      - name: Validate tag matches version.clj
        id: validate
        run: |
          TAG="${GITHUB_REF#refs/tags/v}"
          FILE_VERSION=$(bb -e "(require 'clef.version) (print clef.version/version)")
          if [ "$TAG" != "$FILE_VERSION" ]; then
            echo "Tag version ($TAG) does not match version.clj ($FILE_VERSION)"
            exit 1
          fi
          echo "version=$TAG" >> "$GITHUB_OUTPUT"

      - name: Verify ./clef is up to date with src/
        run: |
          bb gen-script
          git diff --exit-code -- clef

      - name: Smoke-test the script
        run: ./clef --version

      - name: Fetch the GitHub archive zip and compute SHA256
        id: sha
        env:
          VERSION: ${{ steps.validate.outputs.version }}
        run: |
          URL="https://github.com/kirahowe/clef/archive/v${VERSION}.zip"
          # The tag was just pushed; wait briefly for GH to make the archive available.
          for i in 1 2 3 4 5; do
            if curl -fsSL -o archive.zip "$URL"; then break; fi
            sleep 5
          done
          sha=$(sha256sum archive.zip | cut -d' ' -f1)
          echo "sha256=$sha" >> "$GITHUB_OUTPUT"

      - name: Checkout tap repo
        uses: actions/checkout@v4
        with:
          repository: kirahowe/homebrew-clef
          token: ${{ secrets.HOMEBREW_TAP_TOKEN }}
          path: tap

      - name: Rewrite Formula/clef.rb
        env:
          VERSION: ${{ steps.validate.outputs.version }}
          SHA: ${{ steps.sha.outputs.sha256 }}
        run: |
          cat > tap/Formula/clef.rb <<EOF
          class Clef < Formula
            desc "Opinionated full-stack web framework for Clojure (pre-alpha)"
            homepage "https://github.com/kirahowe/clef"
            url "https://github.com/kirahowe/clef/archive/v${VERSION}.zip"
            sha256 "${SHA}"
            version "${VERSION}"
            license "MIT"

            depends_on "borkdude/brew/babashka"

            def install
              bin.install "clef"
            end

            test do
              assert_match version.to_s, shell_output("#{bin}/clef --version").strip
            end
          end
          EOF

      - name: Commit and push formula update
        run: |
          cd tap
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add Formula/clef.rb
          git commit -m "clef ${{ steps.validate.outputs.version }}"
          git push
```

- [ ] **Step 2: Commit**

```bash

git commit -m "ci: add tag-triggered Homebrew formula bump workflow"
git push
```

(Don't dry-run with a throwaway tag yet — the validation step requires the tag to match `version.clj`, which is currently `"0.0.1"`. The first real test is Task 9.)

## Execution notes — deviations & decisions (2026-06-07)

The plan shipped as designed. The following diverged from the written steps; recorded here so the next phase starts from reality, not the plan-as-written.

1. **`bb.edn` tasks invoke `clojure`, not `clj`.** The plan specified `clj -X:test` etc. On CI, `clj` is the `rlwrap` wrapper and fails without it (`Please install rlwrap … or use 'clojure' instead`). All `bb.edn` task commands use `clojure` (non-interactive). The committed `bb.edn` reflects this — no `clj` remains.

2. **Code/CI changes landed via squash-merged PRs, not direct commits to `main`.** The later tasks say "Commit" (implying a direct push). Operating policy blocks direct pushes to `main`, so `ci.yml`, `release.yml`, and the changelog date each went through a short-lived branch → PR → `gh pr merge --squash --delete-branch`. Net `main` history is linear and clean; no functional difference.

3. **Private tap must be added by SSH URL, not plain `brew tap`.** Plan Task 9 (Steps 5 & 8) used `brew tap kirahowe/clef`. The repo is private and HTTPS has no credential helper on this machine, so tapping requires `brew tap kirahowe/clef git@github.com:kirahowe/homebrew-clef.git`. This is local-machine-specific and becomes moot when the tap goes public (planned at v1). The release **workflow** is unaffected — it checks out the tap over HTTPS using the `HOMEBREW_TAP_TOKEN` PAT.

4. **Release PAT must grant access to the *tap* repo.** The first release run failed at "Checkout tap repo" with a 404 — the fine-grained PAT had been scoped to the wrong repository. Re-scoped to include `kirahowe/homebrew-clef` (Contents read/write, Metadata read), then reran the job. Same secret value, so `HOMEBREW_TAP_TOKEN` did not need rotating.

5. **Restore-to-0.0.1 used `brew reinstall`, not `brew upgrade`.** Plan Step 8 used `brew upgrade clef` to undo the test bump. `brew upgrade` will not perform a *downgrade* (`0.0.2-test` → `0.0.1`), so the restore required `brew reinstall clef` after the tap formula was reverted.

6. **Upgrade-test version bump rode the tag, not `main`.** To keep `main` pristine (and avoid the direct-push policy), the throwaway `0.0.2-test` bump commit was carried by the `v0.0.2-test` tag rather than pushed to `main`. The release workflow triggers on the tag regardless, so the loop was still exercised honestly. Tag and bump were removed during cleanup.

7. **Cleanup caught an unpushed tap revert.** During teardown, the tap formula revert (back to `0.0.1`) was committed locally but not pushed, so the remote tap and Homebrew's clone kept serving `0.0.2-test` while the local working copy looked correct. Resolved by pushing the revert, fast-forwarding Homebrew's clone, and `brew reinstall`. Final state verified: `clef --version` → `0.0.1`, remote tap at `0.0.1`, no `v0.0.2-test` tag locally or remote.

**Cross-session facts** worth carrying forward are also in agent memory: the SSH-URL tap requirement, the commit-message style (plain subjects, no `feat:/fix:` prefixes), and the Java-via-jabba / borkdude-tap tooling notes.
